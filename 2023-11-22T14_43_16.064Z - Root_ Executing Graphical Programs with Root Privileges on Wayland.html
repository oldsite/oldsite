How to run graphical applications as root under wayland

Question:

With the implementation of the Wayland display server, certain applications which need root access are unable to function. I have devised a solution to work around this problem, but I am open to alternative approaches or feedback.

Numerous solutions have been suggested to address this problem, including the use of adding root to xhost. While this may resolve the issue, it does compromise the Wayland security model to some extent. A preferable approach would be to add root to xhost temporarily while the program (such as gparted or synaptic) is running.

Gufw:

Modify the gufw.desktop file (located at /usr/share/applications/gufw.desktop in Debian 10) by editing the specified line.

Exec=gufw

to

Exec=sh -c "xhost +si:localuser:root && gufw && xhost -si:localuser:root"

Synaptic:

The instructions provided in this post: https://discourse.ubuntu.com/t/adding-applications-to-start-up/9288 can be used as a source of inspiration. In order to modify the behavior of the /usr/bin/synaptic-pkexec file, it is suggested to comment out the zenity warning message (for Debian 10) and to update the specific line accordingly.

exec "/usr/sbin/synaptic" "$@"

to 

xhost +si:localuser:root
pkexec "/usr/sbin/synaptic" "$@"
xhost -si:localuser:root

It is possible that other programs can be modified similarly to gufw. However, this approach does not address the fundamental problem of older programs running all processes, including GUI, as root. Nonetheless, this modification enables these programs to operate as they did previously.


Solution 1:

In case root has ssh enabled, access to waypipe can also be gained through this link: https://gitlab.freedesktop.org/mstoeckl/waypipe. 

waypipe ssh root@localhost synaptic

Solution 2:

Due to the unavailability of the Wayland socket file located at $XDG_RUNTIME_DIR/$WAYLAND_DISPLAY , other users are unable to launch applications in Wayland. Nevertheless, root can easily access the socket file by specifying its complete path in WAYLAND_DISPLAY, disregarding all file permissions. Additionally, defining XDG_RUNTIME_DIR to a value, although not necessarily used, is essential due to an implementation detail in the Wayland client library.

It's feasible to operate Synaptic as a Wayland-based application exclusively by utilizing a method such as. 

sudo /bin/env WAYLAND_DISPLAY="$XDG_RUNTIME_DIR/$WAYLAND_DISPLAY"  XDG_RUNTIME_DIR=/user/run/0  /usr/sbin/synaptic

As of now, I have only tried out that technique with Sway.

How can I run a graphical application in a container, To run Wayland applications in docker without X, you need a running wayland compositor like Gnome-Wayland or Weston. You have to share the Wayland socket. You find it in XDG_RUNTIME_DIR and its name is stored in WAYLAND_DISPLAY.As XDG_RUNTIME_DIR only allows access for its owner, you need the same user in container as on host. Example: docker run -e XDG_RUNTIME_DIR=/tmp \ -e WAYLAND_DISPLAY

However, it does not manage any other functions beyond that and is similar to a basic window manager such as IceWM. To enhance the system, I would employ containers containing a simplified Linux kernel, X server, and application code. This would allow for isolated X applications to run on top of the primary Wayland system.

The problem is my lack of comprehension on how to operate containers devoid of incurring Xwayland's extra burden, as previously stated in the response.

Solution 1:

The comment that was quoted appears to be confused. It is advised to thoroughly examine the subsequent reply, as it resolves the confusion.

You need an X server in any case.

To execute an X application on a Wayland display server, it is necessary to utilize Xwayland as an intermediary for translation.

To fully utilize Wayland's performance and prevent any additional burdens, it is necessary to have containers that are capable of supporting native Wayland containers. One example of this is running Flatpak applications while under Wayland. In contrast, if you use Flatpak apps under an X desktop, it simply gives up and permits complete insecure X access.

Solution 2:

In principle, X is network transparent, which means you have the flexibility to run the X server wherever you want instead of having to run it in every container containing an app. You can run the X server in a separate container linked to the app containers by network, on your host, or even on another computer serving as an X terminal, and it will not make any difference.

To enable networked clients, simply configure the X server and ensure that all app-containing containers have DISPLAY set to the correct address. Additionally, corrected credentials may need to be established for .Xauthory and other related configurations.

Despite the prevalence of modern gui applications that heavily rely on accelerated graphics hardware for visual effects, these applications may not perform optimally over a network due to potential latency issues or inadequate hardware resources on the remote machine.

If that's the scenario, I suggest considering the implementation of a VNC system that includes VirtualGL.
